## üîÑ Sprint Ceremonies

### Sprint Planning (Day 0)
**Duration**: 2-4 hours
**Participants**: All developers
**Goal**: Select user stories, break into tasks, commit to sprint goal

**Agenda**:
1. Review product backlog (30 min)
2. Select user stories for sprint (30 min)
3. Break down into tasks (60-90 min)
4. Estimate tasks (30 min)
5. Commit to sprint goal (15 min)

---

### Daily Standup (Every day)
**Duration**: 15 minutes
**Time**: 10:00 AM (or team preference)
**Format**: Each developer answers:
- What did you complete yesterday?
- What will you work on today?
- Any blockers?

**Guidelines**:
- Keep it brief (max 2 min per person)
- Focus on progress and blockers
- Detailed discussions happen after standup
- Update task board during standup

---

### Mid-Sprint Review (Day 5)
**Duration**: 1 hour
**Participants**: All developers
**Goal**: Check progress, adjust if needed

**Agenda**:
1. Review burndown chart (10 min)
2. Discuss progress (~50% done?) (20 min)
3. Identify risks (15 min)
4. Adjust scope if needed (15 min)

---

### Sprint Review (Day 10 - End of Sprint)
**Duration**: 2 hours
**Participants**: Developers + stakeholders
**Goal**: Demo completed work, gather feedback

**Agenda**:
1. Demo completed user stories (60 min)
2. Gather feedback (30 min)
3. Accept/reject user stories (15 min)
4. Update product backlog (15 min)

**Definition of Done** for User Story:
- All acceptance criteria met
- Code reviewed and merged
- Tests passing
- Deployed to staging
- Documentation updated

---

### Sprint Retrospective (Day 10 - After Review)
**Duration**: 1 hour
**Participants**: All developers (internal)
**Goal**: Reflect on process, identify improvements

**Agenda**:
1. What went well? (20 min)
2. What could be improved? (20 min)
3. Action items for next sprint (15 min)
4. Appreciation round (5 min)

**Retrospective Format** (varies):
- **Sprint 1**: Start-Stop-Continue
- **Sprint 2**: Mad-Sad-Glad
- **Sprint 3**: 4Ls (Liked, Learned, Lacked, Longed for)

## üìã Backlog Prioritization

### Priority Levels

**P0 - Must Have (MVP Blocker)**
- Critical for MVP launch
- No product without this
- Example: User authentication, task creation

**P1 - Should Have (High Value)**
- Important for MVP
- Adds significant value
- Example: Team invites, comments

**P2 - Nice to Have (Medium Value)**
- Enhances product
- Not critical for MVP
- Example: Activity feed, advanced filters

**P3 - Future Consideration (Low Value)**
- Defer to post-MVP
- Nice-to-have features
- Example: Custom themes, export to PDF

## üìä Story Point Guide

### Story Point Estimation

**1 point** - Trivial (1-2 hours)
- Simple UI component
- Basic CRUD endpoint
- Minor bug fix

**2 points** - Easy (2-4 hours)
- Form with validation
- Simple API endpoint with DB
- Basic component with state

**3 points** - Small (4-8 hours)
- Multi-field form with API
- CRUD endpoints for resource
- Component with multiple states

**5 points** - Medium (1-2 days)
- Feature with backend + frontend
- Complex form with validation
- Feature with multiple components

**8 points** - Large (2-3 days)
- Full user story with tests
- Complex feature with multiple APIs
- Feature with real-time updates

**13 points** - Very Large (3-5 days)
- Major feature (should be split)
- Complex integration
- Feature with AI/ML

**21 points** - Epic (5+ days)
- Too large, must be split
- Consider breaking into smaller stories

## üöÄ Getting Started with Sprints

### For Developers

1. **Before Sprint**:
   - Review sprint planning document
   - Understand sprint goal
   - Know your Day 1 tasks

2. **During Sprint**:
   - Attend daily standup
   - Update task board daily
   - Ask for help when blocked
   - Write tests alongside code
   - Keep PRs small and focused

3. **After Sprint**:
   - Demo your work in sprint review
   - Share feedback in retrospective
   - Reflect on what you learned

### For Product Owner

1. **Before Sprint**:
   - Prioritize product backlog
   - Write clear acceptance criteria
   - Be available for clarifications

2. **During Sprint**:
   - Answer questions
   - Review completed work
   - Provide feedback

3. **After Sprint**:
   - Accept/reject user stories
   - Update backlog priorities
   - Plan next sprint
